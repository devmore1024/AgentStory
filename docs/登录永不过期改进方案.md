# 登录永不过期 - 改进方案

## 版本
- 实施日期: 2026-02-16
- 版本: 0.12.8

---

## 问题分析

### 原始问题
用户登录后,每次进来都要重新登录,无法保持登录状态。

### 根本原因

1. **Refresh Token 过期机制**
   - Access Token 有效期: 2小时
   - Refresh Token 有效期: 30天
   - 超过30天不活跃,refresh token 过期 → 必须重新登录

2. **Token 刷新失败直接登出**
   ```typescript
   const refreshed = await refreshAccessToken(user.refreshToken)
   if (!refreshed) return null  // ← 硬失败!
   ```
   任何刷新失败(API 错误、网络问题)都导致立即登出

3. **Token 过期判断过于严格**
   ```typescript
   const isExpired = new Date() >= user.tokenExpiresAt  // ← 刚过期也刷新
   ```
   没有提前刷新的缓冲区,容易遇到边界问题

4. **Cookie Domain 设置缺失**
   - 跨子域名访问时 Cookie 不可见
   - www.agentravel.me 和 agentravel.me 不共享 Cookie

---

## 改进方案

### ✅ 改进 1: Refresh Token 自动轮换

**原理**: 每次刷新 token 时,都保存新的 refresh token

**实现**:
```typescript
// 保存新返回的 refresh_token (如果 API 返回)
const newRefreshToken = refreshed.refresh_token || user.refreshToken

await prisma.user.update({
  where: { id: user.id },
  data: {
    accessToken: refreshed.access_token,
    refreshToken: newRefreshToken,  // ← 关键:更新 refresh token
    tokenExpiresAt: new Date(Date.now() + refreshed.expires_in * 1000),
  },
})
```

**效果**:
- 只要用户持续使用,refresh token 不断更新
- 实现"活跃用户永不过期"
- 只有超过30天不活跃才需要重新登录

---

### ✅ 改进 2: Token 提前刷新(宽限期)

**原理**: 在 token 真正过期前5分钟就刷新

**实现**:
```typescript
const GRACE_PERIOD_MS = 5 * 60 * 1000  // 5分钟
const tokenExpiryTime = user.tokenExpiresAt.getTime()
const currentTime = Date.now()
const isExpired = currentTime >= (tokenExpiryTime - GRACE_PERIOD_MS)
```

**效果**:
- 避免边界条件问题
- 给刷新操作留出时间缓冲
- 用户体验更流畅,不会遇到"刚好过期"

---

### ✅ 改进 3: 刷新失败容错机制

**原理**: 刷新失败时不立即登出,而是给予1小时宽限期

**实现**:
```typescript
if (!refreshed) {
  const EXPIRED_GRACE_PERIOD = 60 * 60 * 1000  // 1小时
  const expiredDuration = currentTime - tokenExpiryTime

  if (expiredDuration < EXPIRED_GRACE_PERIOD) {
    console.warn(`[Auth] Token 刷新失败,但在宽限期内。过期时长: ${Math.round(expiredDuration / 1000)}秒`)
    return user  // ← 允许继续使用旧 token
  }

  console.error(`[Auth] Token 刷新失败且超过宽限期。过期时长: ${Math.round(expiredDuration / 1000)}秒`)
  return null  // 超过宽限期才登出
}
```

**效果**:
- API 临时故障不影响用户使用
- 给予系统自我修复的时间
- 提升鲁棒性,减少误登出

---

### ✅ 改进 4: Cookie Domain 优化

**原理**: 设置正确的 domain,支持跨子域名

**实现**:
```typescript
function getCookieDomain(request: NextRequest): string | undefined {
  const url = new URL(request.url)

  // 生产环境: agentravel.me 或 www.agentravel.me
  if (url.hostname === 'www.agentravel.me' || url.hostname === 'agentravel.me') {
    return 'agentravel.me'  // ← 跨子域名共享 Cookie
  }

  return undefined  // 开发环境不需要 domain
}

response.cookies.set('user_id', user.id, {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  sameSite: 'lax',
  maxAge: 60 * 60 * 24 * 30,  // 30天
  path: '/',
  domain: getCookieDomain(request),  // ← 应用 domain
})
```

**效果**:
- www.agentravel.me 和 agentravel.me 共享登录状态
- 避免跨域访问时 Cookie 丢失
- 支持 CDN/负载均衡场景

## 潜在风险和缓解

### 风险 1: Refresh Token 轮换失败

**场景**: SecondMe API 不返回新的 refresh token

**缓解**:
```typescript
const newRefreshToken = refreshed.refresh_token || user.refreshToken
// ↑ 如果 API 不返回新 token,保留旧的
```

### 风险 2: 1小时宽限期过长

**场景**: Token 长期失效但仍然允许访问

**缓解**:
- 宽限期仅1小时
- 记录警告日志便于排查
- 生产环境可监控频繁触发

### 风险 3: Cookie Domain 设置错误

**场景**: 某些浏览器或代理配置可能阻止

**缓解**:
- 开发环境不设置 domain(undefined)
- 仅在生产环境设置
- 保留 path 和其他安全属性
