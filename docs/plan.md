# AgentStory 下一步改造计划

## 目标

围绕你提出的 3 件事，先做一轮“低风险、可回滚、可验证”的改造：

1. 文案层把“回去”整体转向“冒险”，但保留语义不合适场景，必要时做融合文案。  
2. 首页童话入口收敛到更经典集合：仅保留格林童话与安徒生童话（其余先隐藏）。  
3. 冒险故事生成改为异步触发，避免阻塞主流程与页面跳转。

## 范围与现状定位

- 文案主要分布在首页、书籍页、进入页、记忆页、同行页与导航组件。
- 首页童话数据目前来自 `fairy_tale` 分类 + `isVisibleInPrimaryEntry` 筛选。
- 冒险生成目前在 Server Action -> `continueAdventure` / `publishCompanionFromPersonal` 中同步等待 LLM，属于请求内阻塞链路。

## 执行方案

### 1) 文案改造：回去 -> 冒险（按语义分层）

先建立替换规则，再落地代码，避免机械替换：

- 强替换场景（直接改“冒险”）：
  - CTA、按钮、导航、功能入口（如“继续回去”“一起回去”）。
- 融合场景（改为“回到故事里继续冒险”）：
  - 需要保留“返回故事世界”语义的句子。
- 保留场景（不替换）：
  - 与“回忆/归来/回家”等语义绑定，替换后会失真。

改造方式：

- 先提取页面文案常量（同文件内轻量常量即可），按规则逐一替换。
- 完成后做一次全局检索，确认不出现明显冲突文案。

验收标准：

- 关键入口文案统一表达“冒险感”，但叙事句不生硬。
- 不出现“回去”与“冒险”混用导致语义冲突的同屏文案。

### 2) 首页童话收敛：仅保留格林/安徒生

数据策略分两步，先快后稳：

- 第一步（快速上线）：
  - 在首页童话获取链路追加“白名单过滤”，仅允许格林与安徒生来源书目进入主入口。
- 第二步（可选优化）：
  - 在 fairy catalog 层补充“经典来源标签字段/判定函数”，把白名单从页面层下沉到数据层，减少重复逻辑。

实现要点：

- 复用已有 `isVisibleInPrimaryEntry` 机制，不改其他分类入口。
- 对“不在格林/安徒生”的童话仅隐藏首页入口，不删除数据。

验收标准：

- 首页童话列表仅出现格林/安徒生集合。
- 其他童话在详情页或后台数据仍可保留，不影响后续恢复展示。

### 3) 冒险生成异步化：去阻塞主流程

采用“先入队/标记，再后台生成”的结构，保持现有数据模型可演进：

- 请求侧（Server Action）：
  - 用户触发后立即创建/更新线程状态并返回（redirect 不等待 LLM）。
  - 返回页展示“生成中”状态。
- 任务侧（Worker/调度函数）：
  - 后台消费待生成任务，执行 `generateAdventureEpisodeWithLlm`，成功后写入 episode 与 timeline。
  - 失败写入错误状态，可重试。
- 读取侧（页面）：
  - 根据线程状态渲染“生成中 / 已完成 / 失败重试”。

最小改造路径：

- 先在现有 `story_experience_jobs` 基础上补齐状态流转与触发点，不一次性引入新基础设施。
- 先完成“异步触发 + 轮询刷新”版本，再评估 SSE/实时推送。

验收标准：

- 提交动作不再等待 LLM 完成即可跳转。
- 异常时页面可见失败状态，不会默默卡住。
- 成功后线程与章节数据与原同步版本一致。

## 实施顺序

1. 文案替换规则与落点清单确认。  
2. 首页童话白名单过滤上线。  
3. 冒险生成异步化（状态字段 -> 入队触发 -> 后台消费 -> 页面状态）。  
4. 补测试与回归验证（文案快照/数据过滤/异步状态流）。

## 测试与验证计划

- 单测：
  - 首页童话过滤函数：验证仅格林/安徒生保留。
  - 异步任务状态机：pending -> running -> success/failed。
- 集成验证：
  - 提交冒险后立即跳转成功，页面出现“生成中”。
  - 后台完成后可看到新章节；失败时出现可理解提示。
- 回归检查：
  - 首页、书籍页、进入页、同行页关键文案一致性。

## 风险与回滚

- 风险：
  - 白名单判定来源字段不稳定，可能误隐藏。
  - 异步化初期可能出现“任务已创建但未消费”。
- 缓解：
  - 白名单采用显式 slug/source 双重判断。
  - 增加任务健康检查与失败可重试入口。
- 回滚：
  - 文案与过滤均可按提交粒度回滚。
  - 异步链路保留同步兜底开关（仅内部配置开启）。
